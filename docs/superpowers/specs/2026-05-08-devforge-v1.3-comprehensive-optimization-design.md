## Purpose
Standardize the precondition check pattern across all DevForge skills.

## Template

```markdown
## Precondition Check

Before starting, read `PROJECT_SCAFFOLD/docs/architecture/system/STATE.md`.
- Acceptable phases: `{phase_list}`

- `{minimum_phase}`: the earliest acceptable phase
- `{prerequisite_skill}`: the skill to recommend if preconditions are not met
- `{required_artifact}`: the artifact that must exist (e.g., `architecture.xml`)

### 5.1 security-audit 纳入 Stage Flow

**定位**：在 design-review 之后、project-scaffolding 之前自动触发（可选但推荐）。

**修改后的 Stage Flow**：

```
1. requirement-analysis (Design)
    ↓ [APPROVE]
2. architecture-design (Design deepen)
    ↓ [APPROVE]
    ┌──────────────────┬──────────────────┐
    ↓                  ↓                  ↓
3a. architecture-    3b. design-review   3c. security-audit (NEW)
    validation          (adversarial)       (security scan)
    (technical)         inspection          (optional)
    ↓ [APPROVE]         ↓ [APPROVE/FIX]     ↓ [APPROVE/SKIP]
    └──────────────────┴──────────────────┘

4. project-scaffolding (Implement)
    ↓ [APPROVE]
5. module-design (Design + Implement)
    ...
```

**触发条件**：
- design-review 完成后自动提示："设计审查已完成。是否运行安全审计？回复 `[SECURITY_AUDIT]` 运行，或 `[SKIP]` 跳过。"

**前置条件**：`design_review_completed`

### 5.2 iteration-planning 前置条件收紧

修改前置条件检查：

```markdown
## Precondition Check
- Acceptable phases: `module_design_completed`, `iteration_planning_completed`
- If phase is `scaffolding_completed`:
  - Analyze iteration scope
  - If scope contains `add_module` or `modify_coupling`:

  - If scope contains only `extend_interface` or `no_impact`:
    - Allow proceeding with warning: "当前仅 module_design 部分完成，迭代范围限于已有模块的接口扩展"
```

### 5.3 validation 与 design-review 职责边界澄清

在 `CLAUDE.md` 中新增职责对比图：

```markdown
## Stage 3: 双重验证机制

| 维度 | 3a. architecture-validation | 3b. design-review |
|------|---------------------------|-------------------|
| **目的** | 验证"设计是否被正确实现" | 验证"设计是否有遗漏/错误" |
| **视角** | 技术一致性（工程师视角） | 对抗性审查（批评者视角） |
| **输入** | architecture.xml, INTERFACE_CONTRACT.md | PRD, architecture.xml, DECISION_LOG |
| **检查项** | XML Schema 合规、接口一致性、PRD 可追溯性 | 安全性、可运维性、可扩展性 |
| **输出** | VALIDATION_REPORT.md（PASS/FAIL） | DESIGN_REVIEW.md（问题清单，无 PASS/FAIL） |
| **结果** | 失败必须修复后才能继续 | 问题可接受、推迟或修复 |
| **类比** | 编译器类型检查 | 代码审查 |

**两者关系**：validation 确保"设计文档自洽"，design-review 确保"设计决策正确"。两者互补，不可互相替代。
```

## 6. 工作包E：细节完善（P2）

### 6.1 并行 Batch Mode 协调模式详细步骤

补充 module-design SKILL.md 中的协调模式：

```markdown
### Coordination Mode (Conflict Resolution Detail)

When conflicts are found after parallel subagents complete:

1. **Conflict presentation**: Present a conflict matrix to user:
   ```
   | Conflict ID | Type | Module A | Module B | Description | Severity |
   |-------------|------|----------|----------|-------------|----------|
   | C-001 | interface_mismatch | OrderService | InventoryService | Output schema "order-created" ≠ Input schema "order-received" | blocking |
   | C-002 | state_conflict | UserService | AuthService | State "session_token" has different lifecycle definitions | warning |
   ```

2. **User decision**: For each conflict, user chooses:
   - `[FIX {conflict_id}]` — Fix now (enter fix sub-mode)
   - `[DEFER {conflict_id}]` — Mark as known issue, continue
   - `[ROLLBACK]` — Roll back to serial mode and redesign affected modules

3. **Fix sub-mode**:
   - Read both modules' artifacts
   - Generate proposed fix (e.g., update interface schema, rename state entry)
   - Write fix diff to `DESIGN_REVIEW_FIX_{conflict_id}.md`
   - Ask user: "修复方案已生成。回复 [APPLY] 应用，[EDIT] 手动调整，[IGNORE] 接受风险。"

4. **Post-fix verification**: After all blocking conflicts are resolved, re-run consistency check.

5. **Completion**: If no more blocking conflicts, update STATE.md and proceed to human gate.
```

## 9. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| 路径变更破坏现有用户项目 | 高 | 在 CHANGELOG.md 中明确标注 BREAKING CHANGE；提供迁移脚本 |
| 工具规范提取导致技能文件空洞 | 中 | 保留技能特定的检查点，只提取真正通用的模式 |
| HARD-GATE 补充后流程变僵化 | 中 | 可选阶段（visualization、ops-ready）仍不添加 HARD-GATE |
| Quality Gates 配置增加用户认知负担 | 低 | 提供合理的默认值，用户不配置时自动使用默认值 |
| security-audit 纳入流程后增加耗时 | 中 | 保持可选，用户可 `[SKIP]` |
