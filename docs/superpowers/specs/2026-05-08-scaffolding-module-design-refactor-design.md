# DevForge Scaffolding / Module-Design 流程优化设计

> **Design Date**: 2026-05-08
> **Scope**: `devforge-project-scaffolding` + `devforge-module-design` 职责重组
> **Status**: Approved

---

## 1. 背景与问题

在 DevForge v1.3 中，`devforge-project-scaffolding`（阶段5）和 `devforge-module-design`（阶段6）存在以下核心冲突：

### 1.1 时序悖论（最严重）

- **scaffolding 第4步**声称：基于 `component-spec.xml` 精确生成代码骨架（函数签名、错误处理、文件路径严格匹配）
- **module-design 第6步**才生成：`component-spec.xml`

这意味着 scaffolding 在运行时，`component-spec.xml` 尚不存在，"XML-driven code generation" 的承诺无法兑现。

### 1.2 测试文件双重生成

- scaffolding 生成测试目录 + 基础测试框架
- module-design 生成具体测试文件

两个阶段在测试生成上职责重叠，存在浪费。

### 1.3 模块 vs 组件粒度错位

- scaffolding 按**模块**生成代码文件
- module-design 将模块分解为 3-6 个**组件**

scaffolding 生成的"模块文件"与 module-design 定义的"组件"之间缺乏明确映射。

### 1.4 STATE.md 前置条件不一致

- scaffolding 接受 `module_design_completed` 作为合法前置状态
- module-design 严格要求 `scaffolding_completed`

在标准流程中，scaffolding 不应在 module-design 之后运行。

---

## 2. 目标

1. **消除时序悖论**：`component-spec.xml` 必须在代码生成之前存在
2. **明确职责边界**：每个阶段只做一件事，且做到最好
3. **保持 backward compatible**：现有用户流程不受破坏
4. **兑现 XML as Authority 承诺**：代码骨架必须严格匹配 `component-spec.xml`

---

## 3. 方案：职责重组（方案A）

### 3.1 核心原则

> **scaffolding = 纯基础设施生成器**
> **module-design = 详细设计 + 精确代码骨架生成**

### 3.2 devforge-project-scaffolding 精简

#### 移除的职责

| 原步骤 | 移除内容 | 原因 |
|:---|:---|:---|
| 第4步 | 基于 `component-spec.xml` 的精确代码生成 | XML 尚不存在 |
| 第4步 | P0/P1/P2 代码生成规则 | 无 component-spec 无法实施 |
| 第4步 | "Generate core module files with class/function signatures" | 签名应在 module-design 后生成 |
| 第9步 | 生成具体测试用例 | 测试应在 module-design 后基于模块需求生成 |

#### 保留的职责

| 步骤 | 内容 |
|:---|:---|
| 第1步 | 读取所有历史产物 |
| 第2步 | 内部规划（文件、目录） |
| 第3步 | 生成项目目录树到 `PROJECT_SCAFFOLD/` |
| 第3步 | 生成架构文档树到 `docs/architecture/` |
| 第3步 | 生成依赖配置文件（package.json、requirements.txt 等） |
| 第3步 | 生成 `docs/architecture/INDEX.md` |
| 第3步 | 生成 `repo-index.md` |
| 第6步 | 生成部署拓扑（docker-compose.yml / K8s manifests） |
| 第7步 | 生成 CI/CD 配置（含架构一致性检查 job） |
| 第8步 | 生成 `.env.template` |
| 第9步 | 生成测试**目录结构**（tests/mock/、tests/real/、tests/end_to_end/）+ 框架配置（conftest.py、pytest.ini 等） |
| 第10步 | 生成 `docs/sync-rules.md` |
| 第11步 | 生成 ADR |
| 第12-15步 | 决策日志、自验证、可追溯性审计 |
| 第16步 | 更新 STATE.md |
| 第17步 | Human gate |

#### 修改后的第4步（原"核心代码骨架"）

改为**"模块目录初始化"**：

- 读取 `architecture.xml` 中的 `Module` 列表
- 为每个模块创建空目录（如 `src/modules/{module_id}/`）
- 仅生成 `__init__.py`（或语言等价物）+ 模块头注释（引用 PRD 和 DecisionTrace）
- **不生成**任何函数签名、类定义或业务逻辑占位符
- 模块头注释模板：
  ```python
  # Module: {module_id}
  # Priority: {P0/P1/P2}
  # PRD Reference: PRD.md::Functional Requirements::{req_id}
  # Status: scaffolding-completed (awaiting module-design)
  # Architecture Decision: {DecisionTrace ID}
  # Known Risk: {DESIGN_REVIEW issue IDs if applicable}
  #
  # NOTE: This module's component decomposition and interface contracts
  # will be generated by devforge-module-design. Do not implement
  # business logic until module-design is approved.
  ```

#### 修改后的第9步（原"透明测试夹具"）

改为**"测试框架初始化"**：

- 生成 `tests/mock/`、`tests/real/`、`tests/end_to_end/` 目录
- 生成框架配置文件（conftest.py、pytest.ini、jest.config.js 等）
- 生成测试基类/工具函数（如 MockLLMClient、TestDataFactory）
- **不生成**任何具体测试用例（这些在 module-design 后生成）
- 添加 `tests/README.md` 说明测试组织规则

### 3.3 devforge-module-design 增强

#### 新增第8步：精确代码骨架生成

在现有第7步（测试设计）和第7.5步（测试代码生成）之后，新增：

**第8步：组件代码骨架生成**

- 读取刚生成的 `component-spec.xml`
- 为每个组件生成精确代码骨架：
  - 函数签名 **严格匹配** `ComponentSpec/Functions/Function/Signature`
  - 错误处理 **覆盖** 所有 `ComponentSpec/Functions/Function/ErrorHandling/Error` 条目
  - 文件路径 **匹配** `ComponentSpec/Metadata/FilePath`
  - 添加内联注释引用 XML 节点 ID（如 `# Implements component-spec.xml::AuthController::login`）
- 按组件优先级生成不同深度的代码：
  - **P0 组件**：生成完整接口存根 + 最小实现（非空函数体，返回合理的默认值）
  - **P1 组件**：生成接口存根 + `raise NotImplementedError("P1: implement per module-prd")`（或语言等价物）
  - **P2 组件**：仅生成文件头注释 + 空函数/类定义
- 代码生成到 `PROJECT_SCAFFOLD/{FilePath}`（覆盖 scaffolding 生成的空 `__init__.py` 或创建新文件）

#### 修改后的第7.5步（测试代码生成）

明确职责：

- 读取 scaffolding 生成的测试目录结构
- 为每个组件生成**具体测试文件**：
  - `tests/mock/{module_id}/{component_id}_test.py`
  - `tests/real/{module_id}/{component_id}_test.py`（含 `skipif`）
- 测试函数名映射到 `module-prd.md` 测试用例 ID
- 每个测试函数头注释引用 component-spec.xml 和 module-prd

#### 保留的职责

第1-6步、第9-11步保持不变。

### 3.4 STATE.md 前置条件同步

#### devforge-project-scaffolding

```yaml
Acceptable phases:
  - architecture_design_completed
  - architecture_validated
  - design_review_completed
  - iteration_planning_completed
Explicitly NOT acceptable:
  - module_design_completed  # 在标准流程中，module-design 在 scaffolding 之后
```

> 例外：在**迭代模式**下，如果用户已通过 `[MODULE]` 完成部分模块设计，需要重新生成基础设施， scaffolding 可以接受 `module_design_completed` 和 `iteration_planning_completed`，但**不重新生成**已有模块的业务代码。

#### devforge-module-design

保持不变：
```yaml
Acceptable phases:
  - scaffolding_completed
  - module_design_completed
```

### 3.5 跨阶段产物流转

```
architecture-design
    └── architecture.xml (Module list)
            ↓
project-scaffolding
    ├── 读取 Module list → 创建空模块目录
    ├── 生成 infrastructure (CI/CD, deploy, deps, .env)
    └── 生成 test framework (dirs, config, base classes)
            ↓ [APPROVE]
module-design
    ├── 读取 Module definition from architecture.xml
    ├── 组件分解 → module-architecture.xml
    ├── 组件接口 → module-interface-contract.md
    ├── 组件规格 → component-spec.xml
    ├── 测试用例设计
    ├── 测试代码生成 → tests/mock/{module_id}/...
    └── 代码骨架生成 → src/modules/{module_id}/...
            ↓ [APPROVE / NEXT MODULE]
    (下一个模块或进入 test-execution)
```

## 5. VCMF 检查点更新

### 5.1 devforge-project-scaffolding（精简后）

| Principle | Checkpoint |
|:---|:---|
| Design as Contract | 目录结构、依赖配置、CI/CD 配置必须可追溯至架构决策 |
| Interface as Boundary | CI/CD 中的架构一致性检查 job 必须验证未来生成的代码与 XML 的匹配性 |
| Reality as Baseline | 测试框架必须支持 mock 和 real 两种模式（即使具体测试用例尚未生成） |
| State as Responsibility | `docs/architecture/` 文档树必须完整反映系统级 StateModel |
| XML as Authority | `docs/architecture/INDEX.md` 必须正确索引所有 XML 产物 |

### 5.2 devforge-module-design（增强后）

| Principle | Checkpoint |
|:---|:---|
| Design as Contract | 模块设计必须追溯至系统级 PRD；代码骨架必须追溯至 component-spec.xml |
| Interface as Boundary | 组件接口契约必须显式定义；生成的代码签名必须匹配契约 |
| Reality as Baseline | 测试用例必须覆盖 happy path、abnormal path、state lifecycle |
| State as Responsibility | ModuleStateModel 必须明确每个状态的 location、owner、consumer、lifecycle |
| XML as Authority | 生成的代码骨架必须严格匹配 component-spec.xml；任何不匹配必须通过自验证拦截 |

## 6. 风险与缓解

| 风险 | 缓解措施 |
|:---|:---|
| module-design 工作量增加 | 已有 `[MODULE_BATCH]` 并行模式支持；代码骨架生成是模板化工作，不增加认知负担 |
| 用户习惯于 scaffolding 生成代码 | Human gate 消息明确说明："脚手架阶段仅生成基础设施，业务代码将在模块设计阶段生成" |
| 迭代模式下重复生成代码 | 代码骨架生成遵循 CRUD-Append 规则：已有文件不覆盖，只增量更新；用户手动修改的部分保留 |
| 模块设计与系统级设计不一致 | 自验证步骤新增："System interface honor" 检查已存在，确保模块不违反系统级边界 |
