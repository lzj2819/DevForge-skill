# DevForge SDLC Skill Chain v1.4 混合架构优化设计

**日期**: 2026-05-11  
**版本**: v1.4  
**设计路线**: 方案 C — 混合架构（核心链增强 + 扩展层按需加载）  
**范围**: P0（安全、微服务、数据） + P1（前端、可观测性、性能）

---

## 1. 总体架构

### 1.1 设计哲学

DevForge v1.4 延续 v1.3 的"核心+扩展"哲学，但在两个维度上深化：

1. **核心链增强**: P0 类能力（安全、微服务分布式、数据平台）是生产系统的**基础设施级能力**，纳入主链默认流程
2. **扩展层按需加载**: P1 类能力（前端工程、可观测性、性能测试）是**专业领域能力**，由特征标签驱动动态加载

### 1.3 与 v1.3 的差异概览

| 维度 | v1.3 状态 | v1.4 变更 |
|------|----------|----------|
| 安全工程 | 代码级扫描 (`security-audit`) + 设计层 Attacker Lens | **+** 系统级威胁建模 (`threat-modeling`) + RBAC/ABAC 权限模型 + 密钥管理 |
| 微服务分布式 | K8s 基础 + 蓝绿/金丝雀发布 | **+** Service Mesh (Istio) + 分布式事务 (Saga) + 分布式追踪 (OpenTelemetry) |

## 2. P0 — 核心链增强

### 2.1 安全工程完整化

#### 2.1.1 `devforge-architecture-design` — Security 节点扩展

**当前 v1.3 Security 节点（极简）:**

```xml
<Security>
  <Authentication type="JWT" issuer="auth-service" audience="api"/>
  <Encryption inTransit="TLS1.3" atRest="AES-256"/>
</Security>
```

**v1.4 扩展后 Security 节点:**

```xml
<Security>
  <Authentication type="JWT" issuer="auth-service" audience="api"/>
  <Authorization model="RBAC">    <!-- 新增 -->
    <Role id="admin" permissions="*"/>
    <Role id="user" permissions="read,write"/>
    <Policy type="resource-based"/>
  </Authorization>
  <ThreatModel>                   <!-- 新增 -->
    <STRIDE>
      <Threat category="Spoofing" module="auth-service" mitigation="mfa"/>
      <Threat category="Tampering" module="api-gateway" mitigation="signature"/>
    </STRIDE>
  </ThreatModel>
  <Encryption inTransit="TLS1.3" atRest="AES-256"/>
  <KeyManagement>                 <!-- 新增 -->
    <Strategy type="HashiCorp-Vault" rotation="90d"/>
  </KeyManagement>
  <Audit>                         <!-- 新增 -->
    <LogEvents>authn, authz, data-access</LogEvents>
    <Retention>2555d</Retention>
  </Audit>
</Security>
```

**架构设计阶段新增工作流步骤（Step 5a）:**

- **Security Architecture Modeling**（在 XML 建模之后）
  - 根据 PRD 中的安全需求，填充 `Security` 子节点
  - 自动生成 `security.xml` 独立文件（供 `security-audit` 和 `threat-modeling` 消费）
  - 算法选择验证：扫描 `Encryption` 节点，若发现 MD5/SHA1/DES 则强制告警
  - 权限模型推断：根据 PRD 中的用户角色描述，自动生成 RBAC 初始结构

#### 2.1.2 新增 `devforge-threat-modeling` Core Skill

| 属性 | 值 |
|------|-----|
| **定位** | Core Skill（可选但推荐） |
| **触发** | `[THREAT_MODEL]` 或 `design-review` 后自动建议 |
| **前置条件** | `design_review_completed` 或 `architecture_validated` |
| **输入** | `architecture.xml`, `PRD.md`, `security.xml` |
| **输出** | `THREAT_MODEL_REPORT.md`（STRIDE 矩阵 + 缓解措施 + 安全测试用例） |

**工作流:**

1. **读取输入**: 加载 `architecture.xml` 所有 Module 和 Interface，`PRD.md` 安全需求，`security.xml` 已定义策略
2. **STRIDE 分析**: 对每个 Module × 6 个 STRIDE 维度生成威胁矩阵
   - Spoofing / Tampering / Repudiation / Information Disclosure / DoS / Elevation of Privilege
3. **风险定级**: Critical / High / Medium / Low（基于影响面 × 利用难度）
4. **缓解措施**: 为每个 High+ 威胁生成具体缓解策略
5. **安全测试用例**: 将缓解措施转化为可执行的测试用例
6. **更新 `security.xml`**: 回填 `<ThreatModel>` 节点

**与现有 skill 的分层关系:**

| Skill | 层级 | 关注点 |
|-------|------|--------|
| `devforge-design-review` | 架构层 | Attacker Lens 发现设计层面安全问题 |
| `devforge-threat-modeling` | 系统层 | STRIDE 结构化威胁建模（新增） |
| `sdlc-security-audit` | 代码层 | 具体代码漏洞扫描 |

#### 2.1.3 `sdlc-security-audit` — 增强自动修复

当前 `security-audit` 已提供代码修复示例，v1.4 增强为**自动化 diff 生成**:

- 对每个 Critical/High 问题，输出可直接应用的 `git diff` 格式修复建议
- 生成 `SECURITY_FIX.patch` 文件，用户可 `[APPLY]` 直接应用
- 应用后自动触发重新审计

### 2.2 微服务分布式深化

#### 2.2.1 `devforge-module-design` — 微服务场景增强

当 `architecture.xml` 的 `type="microservice"` 时，组件分解和代码骨架生成额外输出:

| 新增输出 | 说明 |
|---------|------|
| `grpc/{module_id}.proto` | 从 `component-spec.xml` 的 Interface 自动生成 proto 定义 |
| `config/resilience.yaml` | 熔断器、限流器、重试策略配置（Resilience4j 格式） |
| `config/saga-{module_id}.yaml` | 若涉及分布式事务，生成 Saga 状态机定义 |

**Saga 状态机 XML 模型（新增）:**

```xml
<SagaOrchestration id="order-saga">
  <Step id="reserve-stock" module="inventory-service" 
        compensaction="release-stock" timeout="5s"/>
  <Step id="process-payment" module="payment-service" 
        compensaction="refund" timeout="10s"/>
  <Step id="ship-order" module="logistics-service" 
        compensaction="cancel-shipment" timeout="30s"/>
</SagaOrchestration>
```

**`component-spec.xml` 增强:**

新增 `Resilience` 节点：
```xml
<Resilience>
  <CircuitBreaker threshold="50%" window="10s" timeout="30s"/>
  <RateLimiter maxRequests="1000" window="1m"/>
  <Retry maxAttempts="3" backoff="exponential" initialDelay="100ms"/>
</Resilience>
```

#### 2.2.2 `devforge-ops-ready` — 微服务基础设施增强

**新增生成内容:**

| 输出路径 | 内容 |
|---------|------|
| `infrastructure/istio/` | VirtualService、DestinationRule、Gateway 配置 |
| `infrastructure/otel/` | OpenTelemetry Collector 配置、Jaeger 部署 |
| `infrastructure/vault/` | Vault 策略、动态 secret 配置、密钥轮换策略 |
| `infrastructure/network-policies/` | 服务间零信任网络策略 |

**Istio 配置生成规则（从 architecture.xml 推导）:**

- 每个 `Module/Interface` → 一条 VirtualService route
- `Module/Coupling/DependsOn` → DestinationRule 的 subset 定义
- 全局默认策略：mTLS 强制、超时 30s、重试 3 次
- 金丝雀发布场景：自动生成 TrafficSplit 配置

**OpenTelemetry 配置生成规则:**

- 每个 `Module` 自动生成 sidecar 采集配置
- RED 指标自动注入：Rate / Errors / Duration
- 生成 Jaeger 查询界面 Ingress 规则
- 与现有 Prometheus 配置合并输出

### 2.3 数据平台集成

#### 2.3.1 新增 `devforge-data-pipeline` Core Skill

| 属性 | 值 |
|------|-----|
| **定位** | Core Skill（数据项目时触发） |
| **触发** | `[DATA_PIPELINE]` 或 PRD 标签包含 `data_pipeline`/`etl`/`streaming` |
| **前置条件** | `architecture_design_completed` |
| **输入** | `architecture.xml`, `PRD.md`, `DataModel` |
| **输出** | `dataflow.xml`, ETL DAG, 维度建模 Schema, 数据质量规则 |

**与原 `extensions/data-pipeline-design/` 的关系:**

- 原 extension **降级为 reference-only**（提供评估维度、反模式、最佳实践参考）
- `devforge-data-pipeline` skill **接管所有生成逻辑**，纳入主链 phase map
- 原 extension 的 `When to Load` 条件迁移到 skill 的触发条件

**工作流:**

1. **数据流拓扑建模**（Step 1-2）
   - 读取 PRD 中的数据源、转换、存储、消费描述
   - 识别 Source / Ingestion / Transform / Storage / Consumption 节点
   - 输出 `dataflow.xml`：
     ```xml
     <DataFlow>
       <Source id="user-events" type="kafka" format="json"/>
       <Ingestion id="kafka-connect" strategy="cdc"/>
       <Transform id="enrich-user" type="stream" engine="flink"/>
       <Storage id="user-warehouse" type="clickhouse" model="star-schema"/>
       <Consumption id="analytics-dashboard" type="bi-query"/>
     </DataFlow>
     ```

2. **维度建模 Schema 生成**（Step 3）
   - 识别事实表（Fact）、维度表（Dimension）、汇总表（Aggregate）
   - 输出 `schema-olap.sql`（支持 ClickHouse / Snowflake / BigQuery 方言）
   - 包含：
     - Partition 策略（按时间分区）
     - Clustering 策略（按查询维度排序键）
     - 数据保留策略（TTL / 冷热分层）

3. **ETL DAG 生成**（Step 4）
   - 输出 `dags/{pipeline_id}.py`（Airflow 格式）或 `prefect-flows/`（Prefect 格式）
   - 每个 DAG 包含：
     - 依赖管理（任务间顺序和并行）
     - 失败重试策略（指数退避）
     - 幂等性检查（去重键）
     - 数据质量校验任务（集成 `data-quality-rules.yaml`）

4. **数据质量规则**（Step 5）
   - 输出 `data-quality-rules.yaml`：
     ```yaml
     rules:
       - table: fact_orders
         checks:
           - row_count_delta: {threshold: 0.9}
           - null_rate: {column: user_id, threshold: 0.01}
           - freshness: {column: created_at, max_age: 1h}
           - pii_scan: {columns: [email, phone], masking: hash}
     ```

## 3. P1 — 扩展层按需加载

### 3.1 `extensions/frontend-ui-system-design/`

| 属性 | 值 |
|------|-----|
| **类型** | Extension（按需加载） |
| **加载条件** | PRD 标签包含 `ui-heavy`、`design-system`、`spa`、`pwa` |
| **加载时机** | `devforge-architecture-design` 的 pattern selection 阶段 |

**Overlay Rules:**

| 维度 | 权重 | 说明 |
|------|------|------|
| Component Reusability | 1.2x | 组件可复用程度 |
| Bundle Size Impact | 1.1x | 对首屏加载的影响 |
| State Complexity | 1.1x | 全局状态管理复杂度 |
| Accessibility | 1.0x | WCAG 合规性 |
| Animation Performance | 1.0x | 动画帧率性能 |

**Mandatory Modules（前端项目）:**
- `DesignSystem`：颜色、字体、间距、图标规范
- `ComponentLibrary`：基础组件 + 通用组件 + 业务组件（三层）
- `StateStore`：Redux / Vuex / Pinia / Zustand 架构
- `RoutingLayer`：路由守卫、懒加载策略
- `APIService`：OpenAPI 客户端生成、请求拦截

**新增 XML 节点（`architecture.xml` Module 级别）:**

```xml
<Module id="frontend-app" type="frontend">
  <FrontendProfile>
    <Framework type="react" rendering="csr"/>
    <StateManagement type="zustand"/>
    <ComponentLibrary tiered="true"/>
    <PerformanceTargets fcp="1.8s" lcp="2.5s" cls="0.1" fid="100ms"/>
    <I18n strategy="dynamic-load" locales="en,zh,ja"/>
  </FrontendProfile>
</Module>
```

**生成产物:**
- `frontend/design-system/tokens.json` — Design Token
- `frontend/components/{tier}/{Component}.tsx` — 组件骨架（基础/通用/业务分层）
- `frontend/stores/{store}.ts` — 状态管理骨架
- `frontend/e2e/` — Playwright 测试模板
- `frontend/performance-budget.json` — 性能预算配置（含 LCP/FID/CLS 目标）

### 3.2 `extensions/observability-engineering/`

| 属性 | 值 |
|------|-----|
| **类型** | Extension（按需加载） |
| **加载条件** | PRD 标签包含 `high-observability`、`slc-required`、`enterprise` |
| **加载时机** | `devforge-ops-ready` 之前或并行 |

**新增生成内容:**

| 输出 | 说明 |
|------|------|
| `observability/slo-definitions.yaml` | SLO/SLI/Error Budget 定义 |
| `observability/logging-schema.yaml` | 结构化日志 schema（JSON 模式） |
| `observability/alerting-rules.yaml` | 分级告警规则（P1/P2/P3 + 升级策略） |
| `observability/oncall-schedule.yaml` | On-Call 轮换配置（PagerDuty/Opsgenie 格式） |
| `observability/dashboards-as-code/` | Grafana dashboards 的 Jsonnet/CUE 定义 |

**SLO 定义示例:**

```yaml
slos:
  - name: api-availability
    target: 99.9%
    window: 30d
    slis:
      - metric: rate(http_requests_total{status!~"5.."}[5m])
        denominator: rate(http_requests_total[5m])
    alert_burn_rate:
      - multiplier: 2    # 2% budget in 1 day → page
        window: 1d
      - multiplier: 1    # 5% budget in 6 hours → ticket
        window: 6h
```

**与 `ops-ready` 的分工:**

| 维度 | `ops-ready` | `observability-engineering` |
|------|------------|----------------------------|
| 层级 | 基础设施层 | 应用层 |
| Prometheus | 生成基础 RED 指标规则 | 生成业务自定义指标 + SLO recording rules |
| Grafana | 生成基础系统 dashboard | 生成业务级 SLO dashboard + 燃烧率图表 |
| 告警 | 3 条基础规则 | 分级告警策略 + 升级规则 + 自愈触发 |
| 日志 | 无 | 结构化日志 schema + 采集器配置 |

### 3.3 `extensions/performance-testing/`

| 属性 | 值 |
|------|-----|
| **类型** | Extension（按需加载） |
| **加载条件** | PRD 标签包含 `performance-critical`、`high-traffic`、`latency-sensitive` |
| **加载时机** | `devforge-test-execution` 之后或并行 |

**工作流:**

1. **性能测试用例生成**
   - 从 `PRD.md` 的 NFR 和 `architecture.xml` 的 `Interface` 生成 k6 / Artillery / Locust 脚本
   - 覆盖：负载测试、压力测试、峰值测试、耐久测试

2. **基准数据设定**
   - 根据 `architecture.xml` 的 Module 复杂度推断建议并发数
   - 输出 `performance/baseline-config.yaml`：
     ```yaml
     baseline:
       concurrent_users: 1000
       ramp_up: 5m
       sustained_duration: 10m
       thresholds:
         p99_latency: 200ms
         error_rate: 0.1%
         rps_per_core: 500
     ```

3. **性能回归检测**
   - 输出 `performance/regression-check.yaml`
   - 与 CI 集成：每次构建后自动对比上次基准，偏差 > 10% 则标记 regression

**生成产物:**
- `performance/k6/{scenario}.js` — k6 负载测试脚本
- `performance/artillery/{scenario}.yaml` — Artillery 配置
- `performance/reports/` — 性能报告模板
- `.github/workflows/performance-regression.yml` — CI 工作流

## 4. 集成机制与特征标签系统

### 4.1 特征标签识别器（Tag Detector）

**标签来源:**

1. **PRD 自动提取**: `devforge-requirement-analysis` 在生成 PRD 时，根据用户输入自动标记特征标签
2. **用户显式声明**: 用户在需求阶段主动声明（如"这是一个微前端项目"）
3. **架构设计阶段补充**: `architecture-design` 根据模块分解结果追加标签

**标签体系（v1.4）:**

| 类别 | 标签 | 触发加载 |
|------|------|---------|
| **前端** | `ui-heavy`, `design-system`, `spa`, `pwa`, `micro-frontend` | `frontend-ui-system-design` |
| **数据** | `data-pipeline`, `etl`, `streaming`, `batch`, `olap`, `data-warehouse` | `devforge-data-pipeline` skill |
| **微服务** | `microservice`, `service-mesh`, `distributed-transaction` | `microservices-advanced` overlay |
| **安全** | `high-security`, `compliance-required`, `mfa-required` | `devforge-threat-modeling` |
| **可观测性** | `high-observability`, `slc-required`, `enterprise` | `observability-engineering` |
| **性能** | `performance-critical`, `high-traffic`, `latency-sensitive` | `performance-testing` |
| **AI** | `ai-agent`, `llm-orchestration`, `rag`, `multi-agent` | `ai-agent-design` |

**标签冲突处理规则:**
- `microservice` + `high-traffic` → 同时加载 `microservices-advanced` + `performance-testing`
- `data-pipeline` + `ai-agent` → 同时加载 `devforge-data-pipeline` + `ai-agent-design`
- 同一类别标签互斥时取最高优先级（如 `spa` vs `pwa` → 都加载）

### 4.2 Extension 加载协议

**加载时机**: `devforge-architecture-design` 的 Step 1（Architecture parallel exploration）

**加载流程:**

```
1. 读取 PRD.md 中的特征标签
2. 查询 extensions/ 目录下所有 SKILL.md
3. 匹配 When to Load 条件
4. 对匹配的 extension：
   a. 读取其 Additional Dimensions → 加入评估矩阵
   b. 读取其 Mandatory Modules → 加入 architecture.xml
   c. 读取其 Interface Additions → 加入 INTERFACE_CONTRACT.md
5. 在 DECISION_LOG.md 中记录加载了哪些 extensions
```

**Extension 与 Core Skill 的协作模式:**

| 协作模式 | 描述 | 示例 |
|---------|------|------|
| **Overlay** | 增加评估维度和必选项，不替代核心逻辑 | `ai-agent-design` 增加 AI 维度 |
| **Parallel** | 并行执行，各自输出 | `devforge-data-pipeline` 与 `module-design` |
| **Sequential** | 在前一 skill 之后补充 | `observability-engineering` 在 `ops-ready` 之后 |
| **Optional** | 用户确认后加载 | `frontend-ui-system-design` |

| `[THREAT_MODEL]` | `devforge-threat-modeling` | `design-review` 完成 |

### 4.4 核心链 Phase Map 更新


## 7. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| Skill 数量膨胀导致维护成本上升 | 中 | Extension 采用 reference-only 模式，核心逻辑保持在 Core Skills 中 |
| XML Schema 扩展破坏向后兼容 | 高 | 新增节点使用 `optional` 属性，v1.3 产物仍可解析 |
| 用户认知负担增加（命令太多） | 中 | 新增命令仅在对应标签触发时才在 Human Gate 中展示 |
| 生成产物质量下降（范围扩大） | 中 | 每个新增 skill 必须有 Self-validation 章节，遵循 `validation-engine.md` |
