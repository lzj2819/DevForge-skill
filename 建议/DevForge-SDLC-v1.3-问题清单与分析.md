## 一、用户最新修改已解决的问题

### 1.1 门控提示不完整 —— 已解决

**修改文件**：`devforge-project-scaffolding/SKILL.md` + `devforge-module-design/SKILL.md`

**修改内容**：
- 两个技能的门控提示现在都列出了完整命令列表（含命令说明）
- 增加了自然语言反馈处理逻辑：`If user inputs natural language feedback... treat as modification request`

**效果**：用户在任意阶段都能看到所有可用干预命令，不按命令格式输入也能被正确处理。

---

### 1.2 test-execution 与 module-design 测试职责边界模糊 —— 已解决

**修改文件**：`devforge-project-scaffolding/SKILL.md` + `devforge-module-design/SKILL.md`

**修改内容**：
- Scaffolding Step 9 明确声明：`"Do NOT generate specific test cases at this stage — those are generated by devforge-module-design"`
- Scaffolding 只生成测试目录结构和框架配置（`conftest.py`、`pytest.ini`、mock 工具类）
- Module-design Step 7.5 负责生成具体测试代码

**效果**：测试代码生成的唯一责任方明确为 module-design，消除了职责重叠。

---

### 1.3 scaffolding 与 module-design 代码生成职责重叠 —— 已解决

**修改文件**：`devforge-project-scaffolding/SKILL.md` + `devforge-module-design/SKILL.md`

**修改内容**：
- Scaffolding Step 4 明确声明：`"Do NOT generate any function signatures, class definitions, or business logic placeholders at this stage"`
- Scaffolding 只生成模块目录 + `__init__.py`（含头注释）
- Module-design 新增 Step 8 "Component code skeleton generation"，基于 `component-spec.xml` 生成精确代码骨架
- 代码骨架按 P0/P1/P2 分级生成：P0 有最小实现、P1 有 `NotImplementedError`、P2 只有空定义

**效果**：代码生成的边界清晰——scaffolding 做"目录+配置"，module-design 做"设计+代码骨架"。

### 1.4 subagent 模式依赖 superpowers 插件 —— 已解决

**修改文件**：`devforge-module-design/SKILL.md`

**修改内容**：

**修改内容**：

**效果**：模块级产物已统一写入 PROJECT_SCAFFOLD，但系统级产物（PRD、architecture.xml 等）仍在 `skill/artifacts/`。

## 二、仍未解决的问题

### 2.1 产物存储路径的三重标准 —— P0

**问题描述**：同一项目的产物散落在三个不同的根目录下：

| 阶段 | 产物路径 | 状态 |
|------|---------|------|
| requirement-analysis | `skill/artifacts/PRD.md` | 未解决 |
| architecture-design | `skill/artifacts/ARCHITECTURE.md` | 未解决 |
| architecture-validation | `skill/artifacts/VALIDATION_REPORT.md` | 未解决 |
| design-review | `skill/artifacts/DESIGN_REVIEW.md` | 未解决 |
| project-scaffolding | `skill/artifacts/PROJECT_SCAFFOLD/` | 未解决 |
| module-design | `PROJECT_SCAFFOLD/modules/{id}/` | **已修改** |
| test-execution | `docs/architecture/validation/` | 未解决 |
| ops-ready | `infrastructure/` | 未解决 |
| visualization | `docs/architecture/diagrams/` | 未解决 |

**影响**：
- 技能之间引用产物时需要知道对方的路径约定
- 初始阶段产物在 `skill/artifacts/`，迭代后需要手动迁移到 `docs/architecture/system/`
- 用户无法一眼找到所有产物

**建议**：统一为 `PROJECT_SCAFFOLD/docs/architecture/` 下的分层结构，初始阶段就写入最终位置。系统级产物在 `system/`，模块级在 `modules/`，验证报告在 `validation/`，图表在 `diagrams/`。

### 2.2 前置条件检查模板 —— 每个技能完全重复 —— P1

**问题描述**：所有 12 个技能的第一步都是相同的结构：

```markdown
## Precondition Check
Read `skill/artifacts/STATE.md` (or `docs/architecture/system/STATE.md`).
- Acceptable phases: `[phase1]`, `[phase2]`, ...
- If phase is earlier than `[X]`, stop and instruct the user...
- If `architecture.xml` is missing, stop and instruct...
```

**重复度**：100%（12/12 技能）

**建议**：提取为 `skill/tools/precondition-checker.md` 通用规范，每个技能只需引用。

---

### 2.3 Language Adaptation 规则 —— 每个技能完全重复 —— P1

**问题描述**：11/12 个技能都包含完全相同的 Language Adaptation 段落。

**建议**：放入 `references/system-prompt-template.md` 作为全局注入，单个 SKILL.md 中删除。

---

### 2.4 VCMF Checkpoints 表 —— 结构重复，内容高度相关 —— P1

**问题描述**：每个技能都有 4-5 行的 VCMF 表：
- `Design as Contract` 在 requirement-analysis、architecture-design、module-design 中反复出现
- `Interface as Boundary` 在 architecture-design 和 module-design 中重复
- `State as Responsibility` 在 architecture-design 和 module-design 中重复

**建议**：VCMF 表保留，但增加"继承自"标注，明确当前技能的检查点是上级检查点的细化还是新增。

---

### 2.5 "读取父上下文"步骤 —— 高度重复 —— P1

**问题描述**：几乎所有下游技能都读取同一组文件（PRD + architecture.xml + INTERFACE_CONTRACT + DECISION_LOG + STATE.md）。

| 技能 | 读取的文件数 |
|------|-------------|
| architecture-validation | 4-5 |
| design-review | 5 |
| project-scaffolding | 8+ |
| module-design | 5 |
| iteration-planning | 6+ |

**建议**：定义一个标准的"Parent Context Bundle"概念，skills 引用即可，不必每次列举。

---

### 2.6 自验证步骤模式 —— 结构重复 —— P1

**问题描述**：每个技能都有"Self-validation"步骤，模式完全一致：读取产物 → 运行检查规则 → 标记问题 → 修复。

**建议**：抽象为 `skill/tools/validation-engine.md`，定义通用的验证框架（输入产物 + 规则集 → 报告）。

---

### 2.7 状态更新步骤 —— 几乎完全相同 —— P1

**问题描述**：每个技能都执行：Append to Completed Steps → Update Current State → Update Module Registry → Append Known Pitfalls。

**建议**：抽象为通用状态更新协议，skills 只需提供具体值。

### 2.8 HARD-GATE 使用不一致 —— P1

**问题描述**：

| 技能 | 有 HARD-GATE |
|------|-------------|
| requirement-analysis | ✅ |
| architecture-design | ✅ |
| architecture-validation | ✅ |
| design-review | ✅ |
| project-scaffolding | ✅ |
| module-design | ❌ |
| test-execution | ❌ |
| iteration-planning | ❌ |
| visualization | ❌ |
| ops-ready | ❌ |
| debug-assistant | ❌ |
| sdlc-security-audit | ❌ |

**影响**：后半段技能没有 HARD-GATE，暗示它们"可跳过"。但 ops-ready 生成生产基础设施配置，跳过可能导致部署失败；test-execution 没有 HARD-GATE，但测试失败理论上应该阻止继续。

**建议**：为所有"非可选"阶段添加 HARD-GATE，明确哪些阶段是可选的（如 visualization、security-audit）。

---

### 2.9 sdlc-security-audit 的"孤儿"定位 —— P0

**问题描述**：
- CLAUDE.md 的 stage table 中没有它
- 没有明确的触发时机（只在 `.env.example` 中提到 `[SECURITY_AUDIT]` 命令）
- 没有前置条件检查（不读取 STATE.md phase）
- 没有与其他技能的集成点

**影响**：它像是一个独立工具，但 DevForge 的设计哲学是"Workflow-Aggregate Decomposition"——每个技能都是流程的一部分。security-audit 游离在外。

**建议**：要么明确纳入 stage flow（如在 design-review 后自动触发），要么降级为工具规范而非独立 skill。

---

### 2.10 context-compression 的定位矛盾 —— P1

**问题描述**：
- 它是一个独立的 `context-compression/SKILL.md`
- 但触发条件是"由其他技能自动调用"
- 没有人类门控
- 用户不能（也不应该）直接触发它

**建议**：降级为 `skill/tools/context-compression.md` 工具规范，由其他 skill 在工作流中引用。

---

### 2.11 iteration-planning 的前置条件过于宽松 —— P1

**问题描述**：
- 接受 `scaffolding_completed` 作为进入条件
- 但 iteration-planning 可能涉及 `add_module`、`modify_coupling`
- 如果 module-design 还没完成，新增模块的影响分析缺少模块级设计参考

**建议**：将 `add_module` 类型的迭代强制要求 `module_design_completed` 前置条件，或至少检查 Module Registry 的完整性。

---

### 2.12 design-review 与 architecture-validation 的职责边界模糊 —— P1

**问题描述**：两者都检查"架构质量"，只是角度不同：
- architecture-validation：技术一致性（XML Schema、接口一致性、PRD 可追溯性）
- design-review：对抗性审查（安全性、可运维性、可扩展性）

**影响**：用户可能困惑为什么需要两个验证阶段。

**建议**：在 CLAUDE.md 中增加更清晰的职责对比图，明确 3a 是"技术验证"，3b 是"对抗性审查"。

---

### 2.13 覆盖率阈值等参数硬编码 —— P2

**问题描述**：
- test-execution：覆盖率 `< 80%` 标记失败
- project-scaffolding：覆盖率阈值 80%
- ops-ready：Prometheus 告警阈值固定（错误率 >1%，P99 >500ms）

**影响**：一刀切，不适合所有项目类型。

**建议**：在 STATE.md 中增加 `Quality Gates` 配置段，允许用户自定义阈值。

---

### 2.14 debug-assistant 的输入假设过强 —— P2

**问题描述**：Mode A（Bug 诊断）假设输入是测试输出 + 错误日志 + 源码 + component-spec.xml。如果用户从生产环境日志中发现 bug，没有测试输出，Mode A 的第一步就缺失了。

**建议**：增加 Mode C（生产问题诊断），以日志和监控数据为主要输入。

---

### 2.15 并行 Batch Mode 的冲突解决机制过于简单 —— P2

**问题描述**：module-design 的 Parallel Batch Mode 中：如果 subagent 之间产生冲突，"enter coordination mode — fix conflicts sequentially"，但没有定义"协调模式"的具体工作流。

**建议**：补充"协调模式"的详细步骤（展示冲突矩阵 → 用户选择修复策略 → 顺序修复 → 重新验证）。

---

### 2.16 编号和格式不一致 —— P2

**问题描述**：

| 技能 | 编号问题 |
|------|---------|
| requirement-analysis | 步骤 3.5 后直接跳 4，中间无 3.6 |
| architecture-design | 两个"步骤 6"（DDL 和 OpenAPI），后续 7.1 应为 8.1 |
| module-design | 步骤 7 后是 7.5 再是 8，编号跳跃 |
| iteration-planning | 步骤 8.5（应为 9，后续顺延） |
| test-execution | 只有 8 步，但 Gate 是第 8 步（没有 Human gate 标题） |

**建议**：统一使用整数编号，子步骤用字母后缀（如 Step 7a、Step 7b）。

## 三、新发现的问题

### 3.1 scaffolding 门控提示残留旧文本 —— P2

**位置**：`devforge-project-scaffolding/SKILL.md` 第 196 行

**问题**：门控提示中同时存在旧版本和新版本文本：
```markdown
Say exactly: "项目基础设施已生成...回复 [APPROVE] 进入模块详细设计阶段，或提出修改意见。"
Then list all available commands:
  - [APPROVE] — 批准并继续（进入模块详细设计阶段）
  ...
```

"回复 [APPROVE]...或提出修改意见"是旧版本残留，与下方完整命令列表重复。

**建议**：删除 `"回复 [APPROVE] 进入模块详细设计阶段，或提出修改意见。"`，保留 `"请确认当前阶段输出。"` + 命令列表。

---

### 3.2 module-design 的 `__init__.py` 覆盖问题 —— P2

**位置**：`devforge-module-design/SKILL.md` Step 8

**问题**：scaffolding 生成的 `__init__.py` 会被 module-design 的代码骨架覆盖。文档中说 `"overwriting scaffolding-generated __init__.py or creating new files as needed"`，但这是否会产生不必要的文件覆盖操作？

**建议**：明确 scaffolding 生成的 `__init__.py` 应该只包含最小内容（空文件或简单头注释），以便 module-design 无缝覆盖而不丢失信息。

---

### 3.3 module-design 的 Step 8 与 component-spec.xml 的时序 —— P2

**位置**：`devforge-module-design/SKILL.md` Step 6 + Step 8

**问题**：Step 6 生成 `component-spec.xml` 模板，Step 8 基于 `component-spec.xml` 生成代码骨架。但自验证要求 "Generated code skeletons must strictly match component-spec.xml signatures"——如果代码骨架生成过程中修改了签名，两者就会不一致。

**建议**：明确 Step 8 的代码骨架必须**严格基于** Step 6 的 component-spec.xml，不能擅自修改签名。任何修改都应该先更新 component-spec.xml，再重新生成代码骨架。
